## 今日やったこと
- 認証周りの基礎知識をゼロから整理した
- Cookie・セッション・パスワードマネージャの責任境界を言語化した
- PO への回答として「パスワード管理とセッション管理を分ける設計判断」を言葉にした

---

## 決めたこと

パスワード・メールの保存と自動入力は **ブラウザのパスワードマネージャに完全委譲** する。
TETOMI 側の責務は `autocomplete` 属性で正しいヒントを渡すことだけ。

ログイン状態の維持（セッション管理）は **TETOMI サーバー側の責務** として設計する。
Cookie に `HttpOnly` 属性をつけ、JavaScript から読めないようにする。

---

## 理解できたこと

### 認証の 3 段階フロー

```
① メール + パスワードでサーバーが本人確認
② サーバーがセッション ID を発行 → Cookie に保存
③ 以降のリクエストごとに Cookie を自動送信 → サーバーが検証
④ ログアウト → Cookie とサーバーのセッションを両方削除
```

### Cookie の性質
- 保存場所はブラウザ（クライアント）側
- リクエストのたびにブラウザが自動で送信する
- `HttpOnly` をつけると JavaScript から読めなくなる → XSS による盗難を防ぐ

### パスワードマネージャとセッション管理の違い

| | 管理するもの | 使うタイミング |
|---|---|---|
| パスワードマネージャ | メール・パスワード（資格情報） | ログイン時（① のみ） |
| セッション管理 | セッション ID（認証済みの証明） | ログイン後（② ③ ④） |

面接で言える言葉：
> 「パスワードの保存と自動入力はブラウザのパスワードマネージャに委譲しています。TETOMI 側の責務は `autocomplete` 属性で正しいヒントを渡すことだけです。ログイン状態の維持はセッション管理として TETOMI 側で設計しています。この 2 つを分けている理由は、ログアウト時にセッションだけ削除することで、セキュリティを保ちながら UX を損なわない設計にするためです。」

---

## 悩んだこと・トレードオフ

| 選択肢 | メリット | 採用判断 |
|---|---|---|
| ログアウト時にセッションのみ削除 | UX を保ちつつセキュリティ確保 | 採用 |
| ログアウト時に両方削除 | セキュリティは高い | 不採用（毎回パスワード入力が必要になる） |
| ログアウト時に両方残す | 手間がない | 不採用（なりすましリスクが高い） |

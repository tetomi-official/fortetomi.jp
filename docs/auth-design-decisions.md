## 2. 決まったこと

### 大学メール認証の方針

- **在籍担保が最優先**。自己申告フォールバックは認めない。
  - 「メールアドレスを所有していること」と「大学に在籍していること」は別物。
    大学名を自己入力させた瞬間に在籍担保が崩れるため、フォールバックは採用しない。

### 上記から自動的に決まる帰結

- **ドメインのホワイトリスト管理が必須**（許可する `@xxx.ac.jp` のリストをどこかに持つ）。
- **メール確認リンクの送信が必須**（用途は下記「3. PO の質問への回答」を参照）。
- **卒業生はメール失効によりログイン不可になる**（ただし "排除" の正確な定義は論点4で未確定）。
- **編入生・院生・職員** など端ケースは別途検討が必要（「6. 既知の端ケース」を参照）。

## 3. PO の質問への回答（責務境界の整理）

### Q. メール認証・確認リンク送信はできるか、どの場面で使うか

在籍担保を最優先とした方針の結果として、用途は自動的に確定する。

| 用途 | 必須 / 任意 | 理由 |
|---|---|---|
| 初回登録時の在籍 + 本人確認 | 必須 | これがないと「ドメインを信用する」前提が成立しない |
| パスワードリセット | 必須 | メール所有を継続的に担保する唯一の手段 |

確認リンクの仕組み（Supabase Auth を使う場合）:

```
ユーザー        ブラウザ(JS)      Supabase Auth     メールサーバー
   | メアド入力      |                  |                 |
   |--------------->| signUp()         |                 |
   |               |----------------->| 確認メール送信依頼 |
   |               |                  |---------------->|
   |  「メール確認して」表示            |                 |
   |<--------------|                  |                 |
   | (メールの確認リンクをクリック)      |                 |
   | /auth/confirm?token=xxx          |                 |
   |--------------->| verifyOtp(token)|                 |
   |               |----------------->|                 |
   |  ログイン完了   |<-----------------|                 |
```

- 「確認リンク」の正体は **URL に埋め込まれた JWT トークン**。
- ブラウザが `/auth/confirm?token=xxx` を開くと、Supabase がトークンを検証してユーザーを有効化する。
- メール送信自体は Supabase 側のサーバーが行うため、自前のメールサーバーは不要。

### Q. パスワード・メールの保存、自動入力は誰が設定するか

責務を「資格情報の記憶」と「ログイン状態の維持」に分けて整理する。

| 責務 | 担当 | TETOMI 側がやること |
|---|---|---|
| パスワード / メールの記憶 | ブラウザのパスワードマネージャ（保存するかはユーザーが選ぶ） | `autocomplete` 属性を正しく付けるだけ |
| 自動入力の挙動 | ブラウザ | 同上 |
| ログイン状態の維持 | TETOMI サーバー側 | セッション ID を発行し Cookie（`HttpOnly`）で管理 |

`autocomplete` の付け分け例:

- ログインのメール欄: `autocomplete="email"`
- ログインのパスワード欄: `autocomplete="current-password"`
- 新規登録のパスワード欄: `autocomplete="new-password"`

ログイン状態維持（セッション）の基本フロー:

```
① メール + パスワードでサーバーが本人確認
② サーバーがセッション ID を発行 → Cookie に保存
③ 以降のリクエストごとに Cookie を自動送信 → サーバーが検証
④ ログアウト → セッションを削除（資格情報はブラウザに残す）
```

- Cookie に `HttpOnly` を付けると JavaScript から読めなくなり、XSS によるトークン盗難を防げる。
- ログアウトは「セッションだけ削除（資格情報は残す）」を採用。毎回パスワード入力を求めずに
  セキュリティを保つための折衷（詳細は devlog 2026-05-25 のトレードオフ表）。

## 4. 未確定の論点（トレードオフ表つき）

### 論点1: ドメインホワイトリストの追加運用モデル

| 案 | 仕組み | メリット | デメリット | 面接での語り |
|---|---|---|---|---|
| A. 運営審査型 | 運営が新大学を承認したらドメイン追加 | なりすましをほぼ排除 / 在籍担保の方針と整合 | スケール速度が運営の処理能力に律速 / 運用負荷が継続 | 「在籍担保を最優先したので運用コストを払ってでも審査型にした」 |
| B. ユーザー自由登録型 | 新ドメインを自動許可 | スケール最速 / 運営介入不要 | 在籍担保の方針と矛盾（任意ドメイン許可で担保崩壊） | 語れない（設計の一貫性が崩れる） |
| C. ハイブリッド | 初期はホワイトリスト、要望をユーザーから受付 → 運営が判断 | 速度と安全性の折衷 / ユーザーニーズを可視化 | 実装が最も複雑（要望フォーム・管理画面が必要） | 「ユーザー要望を取り入れつつ在籍担保も維持する設計にした」 |

→ 在籍担保最優先の方針上、**B は論理的に棄却**。A か C の二択（未決定）。

### 論点2: 性別取得の目的

| 案 | 規約・ポリシーへの記述 | 取得方法 | リスク |
|---|---|---|---|
| A. 学生同士の交流促進をコアに含める | 「学生間コミュニケーションの円滑化のため」と明記 | 必須 / 男・女・その他・回答しない | README のコンセプト（教科書取引）と矛盾しないかの確認が必要 |
| B. 取得をやめる | 記述不要 | 項目自体を削除 | 「会話が捗る想定」が失われる（ただし未検証仮説を捨てるのは妥当） |
| C. 安全配慮目的で限定取得 | 「手渡し時の安全配慮のため、相手の性別表示を希望できる」と限定記述 | 任意 / 表示も任意 | 目的が限定的すぎて形骸化する可能性 |

→ 現状の根拠「会話が捗る想定」は **未検証の仮説** で A の根拠としては弱い。未決定。

### 論点3: 技術スタック

| 領域 | 候補 | 検討メモ |
|---|---|---|
| バックエンド言語 | PHP / Node.js(TS) / Python | PHP は「習熟しているから速い」は弱い根拠。Laravel のエコシステム等、技術的な根拠の言語化が必要 |
| DB / 認証基盤 | Supabase / 自前 PostgreSQL | Supabase は RLS が大学スコープと相性良。関東数十大学・数万ユーザー規模なら問題なし。「自前が必要」と感じる具体要件（複雑な集計・特殊拡張・コスト上限など）が未言語化 |

→ 「PHP を選ぶ技術的根拠」「Supabase をどこまで使うか / 自前が要る具体要件」が未確定。

### 論点4: 卒業生の "排除" の定義

| 意味 | 内容 | 保証する仕組み |
|---|---|---|
| ① ログインできなくなる | 新しい操作はできない | メール失効 + 認証基盤（Firebase/Supabase Auth）で達成 |
| ② DB から削除・無効化される | 過去の出品データも消える | 別途バッチ処理 / 卒業年度フィールドでの無効化など、明示的な実装が必要 |

→ メール失効だけで保証されるのは **① のみ**。DB のレコードは自動では消えない。
どちらを「排除」と定義するかが未確定。

購入希望の「日時提示」UI 比較モック

現状（案A：1日時を提示してYes/No）と、新案（案B：複数候補から選ぶ）のメリデメと画面イメージ。下のコントロールで候補数や時間粒度を変えて検討できます。
このファイルは単体で開けます（ビルド不要）。色やラベルは <style> の :root と HTML を直接編集してください。

1. メリット / デメリット比較

案A：1日時を提示 → Yes/No現状

買い手の体験

  • 入力が最速（1回で完了）。迷いが少ない
  • 外すと「断る／別日程提案」の往復が発生

出品者の体験

  • 判断がシンプル（承認 or 断る）
  • 都合が合わないと逆提案が必要＝1ターン増える

成約まで

  • 初回でズレると往復が増え、離脱・放置が起きやすい
🛠 実装：済（現状）📦 DB変更：なし

案B：複数候補を提示 → 1つ選ぶ新案

買い手の体験

  • 都合の良い枠を先出し→合意しやすい
  • 入力の手間が増える（最低2件は出す心理負担）

出品者の体験

  • 候補から選ぶだけで確定。逆提案が要らない
  • 断る理由が「全部NG」のときだけに減る
  • 選択肢が多いと一覧がやや長くなる

成約まで

  • 1往復で確定しやすい＝成約率↑・放置↓
🛠 実装：中📦 DB変更：候補をJSON/別表で保持

3. たたき台としての推奨

案B（複数候補から選ぶ）を推奨。ただし「最低1件・最大3件」「2件目以降は任意」にして買い手の入力負担を抑えるのが落としどころ。時間粒度は時間帯（午前/午後）がカジュアルで合意が速い（学生同士のキャンパス受け渡し前提）。

段階導入案：① まず案Bの買い手UIだけ出す → ② 出品者の「候補から選ぶ」を出す → ③ 全NG時のみ現状の逆提案フローへ合流。データは既存の reservations に候補配列（JSON列 or 子テーブル reservation_slots）を足す形で、確定時に preferred_* を埋める。

このモックで決めたいこと：候補の最大数 / 時間粒度（帯 or 時刻）/ 「全NG」時の動線（逆提案を残すか）。

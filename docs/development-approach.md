# 開発の進め方・設計手法に対するスタンス

TETOMI 開発で「どこまで作り込み、どの手法をいつ採用するか」の判断軸を記録するメモ。
就活ポートフォリオとして「なぜその進め方を選んだか」を面接で語れるようにするための意思決定ログ。

最終更新: 2026-06-06

---

## 1. 前提認識：開発の進め方には複数の流派がある

- ドメイン駆動設計（DDD）、ドメインオブジェクトの設定、振る舞いの定義 など、
  作り込んだ設計手法がいくつも存在する。
- これらは強力だが、目的は「複雑さを飼いならすこと」であって、それ自体が目的ではない。

---

## 2. 設計手法の使いどころ（自分なりの仮説）

- DDD などの手法は、**学生（ユーザー）が自分のニーズを完全には自覚できていない状況** を
  理解・言語化するためのツールとして有効だと考えている。
  - 「何が欲しいか曖昧」な相手のニーズを、ドメインモデルとして輪郭を与えることで掘り下げられる。
- ただし、これは「必要だと感じたとき」に効いてくる道具であって、最初から全面適用するものではない。

---

## 3. 今回の方針：速度優先で、まずプロダクトを出す

- 背景：**学生環境であり、とにかく早くプロダクトを形にすることが重要**。
- そのため、初手では既存プロジェクトの流用や、企業で使われている本格的な設計を
  そのまま持ち込まない。
- 代わりに、**「本当にこれが必要だ」と感じた局面で、その手法を取り入れる**。

### この方針で予想される痛み（あらかじめ自覚しておく）

- 難しい部分・しんどい部分が出てくる。
- 「ここはカスタマージャーニーをちゃんと作りたい」と感じる場面が出てくる。
  - → そう感じた時点が、作り込み手法を導入する判断ポイントになる。

### 手法を切り替えるトリガー（現時点の言語化）

- **「なんか上手くいかないな」と感じたタイミング** で、調べて別の手法を試す。
- つまり「手法ありき」ではなく「行き詰まり起点」で投資判断する、直感ベースの運用。
- 課題（未整理）：この「上手くいかない感覚」を、後から振り返って語れるように
  具体的な兆候（手戻りの回数 / 触るのが怖いコード / 議論が前に進まない 等）へ
  落とし込んでおくと、面接での再現性が上がる。

---

## 4. 面接で言える言葉（30秒トーク）

> 「DDD のような作り込んだ設計手法も知った上で、最初からは適用しませんでした。
> 学生プロジェクトでまず速度を優先してプロダクトを出すことを重視し、
> ドメインモデルやカスタマージャーニーといった重い手法は『本当に必要だと感じた局面』で
> 導入する方針にしました。手法ありきではなく、複雑さや痛みが顕在化したタイミングで
> 投資判断をするという進め方です。」
